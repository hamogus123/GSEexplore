# Plan: Resolving Bioconductor URL Resolution Errors in Posit Cloud Deployments

## 1. Problem Statement
When deploying R applications that depend on Bioconductor to **Posit Connect Cloud** or using the **Posit Publisher**, the deployment often fails with the following error:
`Could not scan R packages from lockfile: renv.lock, repository Bioconductor 3.21 cannot be resolved to a URL.`

### Root Cause
The automated scanners used by Positron and rsconnect attempt to map repository names found in `renv.lock` to valid HTTP URLs. While names like "CRAN" are standard, the name **"Bioconductor 3.21"** (generated by recent versions of `renv`) is often not recognized by the scanner's internal lookup table, causing a "pre-flight" crash.

## 2. The Solution: Pre-Generated Manifest with Manual Mapping
Instead of allowing the cloud server to scan the project and fail, we generate a `manifest.json` locally. This file acts as an explicit "roadmap" for the server, where we manually define the URL for the problematic repository string.

### Technical Steps

#### Step 1: Create a Manifest Generation Script
We use a dedicated R script to generate the manifest. This script explicitly maps the repository name to its URL in the active R session before calling the manifest writer.

```r
# make_manifest.R
if (!requireNamespace("rsconnect", quietly = TRUE)) install.packages("rsconnect")
if (!requireNamespace("BiocManager", quietly = TRUE)) install.packages("BiocManager")

# 1. Capture the standard Bioconductor repositories
repos <- BiocManager::repositories()

# 2. MANUALLY INJECT THE MAPPING
# This forces the scanner to recognize the string "Bioconductor 3.21"
repos["Bioconductor 3.21"] <- "https://bioconductor.org/packages/3.21/bioc"

# 3. Apply these repositories to the global options
options(repos = repos)

# 4. Write the manifest.json
# This file will now contain the explicit URLs for all Bioconductor dependencies
rsconnect::writeManifest(
  appDir = ".",
  appPrimaryDoc = "app.R"
)
```

#### Step 2: Ensure Annotation Repository Mapping (Optional but Recommended)
For packages like `org.Hs.eg.db`, the scanner may look in the software repository instead of the annotation repository. We ensure these are correctly mapped in the `renv.lock` file by setting the `Repository` field to `BioCann` (the standard name for Bioconductor annotations).

#### Step 3: Deployment Configuration
Once `manifest.json` is generated, the deployment must be configured to use it as the source of truth:
1.  **Commit `manifest.json`** to your repository.
2.  In the **Posit Publisher** UI, ensure `manifest.json` is **checked** in the file list.
3.  **Uncheck `renv.lock`** to prevent the server from attempting a fresh (and likely failing) scan of the lockfile.

## 3. Benefits of this Approach
*   **Bypasses Scanner Bugs:** It circumvents the publisher's inability to resolve specific versioned Bioconductor strings.
*   **Environment Consistency:** The `manifest.json` includes exact hashes for all packages, ensuring the cloud environment perfectly matches the developer's local state.
*   **Reduces Build Time:** The server spends less time "calculating" dependencies because the roadmap is provided upfront.

## 4. How to Adapt for Different Versions
If you are using a different Bioconductor version (e.g., 3.22 for R 4.5.1), simply update the URL in the mapping script:
`repos["Bioconductor 3.22"] <- "https://bioconductor.org/packages/3.22/bioc"`
